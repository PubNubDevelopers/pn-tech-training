Section 1: Publish Fundamentals

  • Real-time fanout to subscribers on a channel
  • Per-publisher ordering guarantees
  • Store behavior and retrieval through history APIs
  • Error and retry patterns for production reliability

What Publish Is (and Is Not)

Use Publish for

  • Chat, alerts, notifications
  • Multiplayer and collaboration updates
  • IoT and control commands

Do not use Publish for

  • Large file transfer (use Files)
  • Long-running request/response workflows
  • Primary database persistence

SDK Publish Example

const result = await pubnub.publish({
  channel: "chat.room123",
  message: {
    type: "chat.message",
    schemaVersion: "1.0",
    eventId: "msg_abc123",
    ts: Date.now(),
    payload: { text: "Hello World!", userId: "user456" }
  }
});

console.log("timetoken:", result.timetoken);

Recommended path for most app code.

REST Publish Anatomy

https://ps.pndsn.com/publish/{pub_key}/{sub_key}/{signature}/{channel}/{callback}/{message}

Required query params:
- uuid
- auth (if Access Manager enabled)
  • Use signature=0 and callback=0 for typical JSON publishes
  • URL-encode JSON payloads for GET
  • Use POST for larger messages to avoid URI limits

Store vs Fire Decision

Use Case Mode Why
Chat messageStoreReplay and catch-up required
Vote submissionStoreAuditability and correctness
Typing indicatorFire / no-storeEphemeral signal only
Analytics ingestFireNo subscriber fanout needed

Limits and Response Model

  • Max payload: 32 KiB hard limit
  • Recommended: under 25-30 KiB
  • Optimal: under 2-5 KiB for latency and throughput
  • Success response: [1, "Sent", "timetoken"]
[1, "Sent", "17069876543210000"]

Robust Retry Pattern

async function publishWithRetry(channel, message, maxRetries = 3) {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      return await pubnub.publish({ channel, message });
    } catch (error) {
      const code = error.status?.statusCode;
      if (code === 403) await refreshToken();
      else if (code === 413) throw new Error("payload too large");
      else if (code === 429 || code >= 500) {
        const backoff = Math.min(1000 * 2 ** attempt, 30000);
        // await delay(backoff + jitter())
      } else throw error;
    }
  }
  throw new Error("publish exhausted retries");
}

Section 2: Message Design

Design contracts that support routing, versioning, dedupe, and scale.

Four Required Message Fields

{
  "type": "domain.action",
  "schemaVersion": "1.0",
  "eventId": "evt_unique_identifier",
  "ts": 1706889600000,
  "payload": { "applicationData": "..." }
}
  • type: routing and Function dispatch
  • schemaVersion: compatibility contract
  • eventId: idempotency and dedupe
  • ts: client clock for tracing and latency

Payload Guardrails

  • Keep payloads flat (1-2 levels)
  • Use consistent camelCase field names
  • Publish minimal required data only
  • Never include secrets or raw credentials
  • Fetch contextual metadata from App Context or backend

Schema Evolution Strategy

ChangeVersionImpact
Add optional fieldMinor (1.0 -> 1.1)Backward-compatible
Rename/remove fieldMajor (1.x -> 2.0)Consumer updates required
Type changeMajorBreaking contract

Deploy multi-version consumers before switching publishers.

Channel Naming Convention

Pattern: [channelType].[channelId]

Examples:
vote-submit.session123
vote-results.session123
chat.room456
inbox.user789
  • Keep depth at 2 levels for Function wildcard binding
  • Use ASCII-safe characters
  • Encode compound identifiers in the second segment

Access Manager Write Permissions

const token = await pubnub.grantToken({
  ttl: 60,
  authorized_uuid: userId,
  resources: {
    channels: {
      [`vote-submit.${sessionId}`]: { read: false, write: true },
      [`vote-results.${sessionId}`]: { read: true, write: false }
    }
  }
});
  • Participants can submit but cannot forge result messages
  • Server publishes authoritative outcomes

Message Design Checklist

  • Required fields present and valid
  • Schema version documented and tested
  • Payload size and sensitivity reviewed
  • Channel follows 2-level naming strategy
  • Access Manager grants least-privilege write access

Section 3: Advanced Publish

Scale and reliability patterns for high-throughput workloads.

Rate Limits and Throttling

  • 429 indicates publish rate exceeded for the current plan/keyset
  • Apply bounded exponential backoff with jitter
  • Use queues to smooth spikes and control producer pressure
  • Track throttle metrics and publish rate over time

Queue + Backoff Example

class PublishQueue {
  constructor(pubnub, mps = 100) {
    this.pubnub = pubnub;
    this.queue = [];
    this.interval = 1000 / mps;
  }
  enqueue(channel, message) { this.queue.push({ channel, message }); }
  async process() {
    while (this.queue.length) {
      const { channel, message } = this.queue.shift();
      await pubnub.publish({ channel, message });
      // await delay(this.interval)
    }
  }
}

Metadata Filtering with meta

await pubnub.publish({
  channel: "events.room123",
  message: { type: "event.bid", eventId: "bid_123", ts: Date.now(), payload: { amount: 500 } },
  meta: { priority: "high", region: "us-west" }
});

pubnub.subscribe({
  channels: ["events.room123"],
  filterExpression: "meta.priority == 'high' && meta.region == 'us-west'"
});

meta is not persisted and remains unencrypted.

Per-Message TTL

  • TTL is minutes and applies only when stored in history
  • Use short TTL for temporary alerts, longer for chat/audit
  • ttl: 0 means use keyset default retention
await pubnub.publish({
  channel: "chat.room123",
  message: chatMessage,
  storeInHistory: true,
  ttl: 1440
});

Encryption with Cipher Key

const securePubNub = new PubNub({
  publishKey,
  subscribeKey,
  userId: "user123",
  cipherKey: "secure-random-32-char-key"
});
  • AES-256-CBC payload encryption
  • Base64 overhead increases payload size by about 33%
  • Effective encrypted payload target is about 24 KiB
  • Functions cannot read encrypted message payloads

Replication and Fire Behavior

Store/Publish

  • Replicates globally across PoPs
  • Delivers to subscribers worldwide
  • Can store in history

Fire (norep: true)

  • Nearest PoP only
  • No regular subscriber delivery
  • Functions and Illuminate only

Optimization Checklist

  • Keep payload minimal and structured
  • Batch and parallelize across channels
  • Serialize per-channel where ordering matters
  • Use Fire for backend-only analytics pipelines
  • Shard hot channels to distribute fanout load
  • Instrument p50/p95/p99 publish latency

Section 4: Publish Integrations

Combining publish with persistence, push, Functions, routing, and security.

Publish + Message Persistence

// Stored message
await pubnub.publish({
  channel: "chat.room123",
  message: chatMessage,
  storeInHistory: true,
  ttl: 1440
});

// Ephemeral message
await pubnub.publish({
  channel: "typing.room123",
  message: typingMessage,
  storeInHistory: false
});

Typical join flow: fetch recent history, then subscribe for live updates.

History on Join Pattern

const history = await pubnub.fetchMessages({
  channels: ["chat.room123"],
  count: 50
});

displayMessages(history.channels["chat.room123"]);
pubnub.subscribe({ channels: ["chat.room123"] });

Use timetoken ranges for replay windows and audits.

Publish + Mobile Push

  • Embed pn_apns and pn_fcm in the message envelope
  • Recommended payload sizing: APNs under 1.5 KB, FCM under 3 KB
  • Register devices to channels before expecting delivery
  • Include deep-link identifiers in push payload data fields

Publish + Functions

Before Publish

  • Validate required contract fields
  • Moderate content and apply rate controls
  • Deduplicate by eventId
  • Enrich with server-generated metadata

After Publish

  • Trigger side effects
  • Webhook calls and downstream notifications
  • Analytics and observability events

Key Takeaways

Design for correctness first, then optimize for scale.

  • Use idempotent event contracts with stable eventId
  • Keep channel topology simple and authorization explicit
  • Integrate persistence, push, and Functions intentionally
  • Measure throughput and tune with queueing and sharding
